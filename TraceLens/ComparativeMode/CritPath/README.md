# Critical Path Analysis for PyTorch Distributed Training

## Overview

This tool performs critical path analysis on PyTorch distributed training traces to identify performance bottlenecks. It constructs a directed acyclic graph (DAG) from execution traces and computes the longest path through the graph, representing operations that determine the overall execution time.

## Architecture Overview

The critical path analysis pipeline consists of five main stages:

```
Input Traces → Link → Prune → Build Dependencies → Create DAG → Find Critical Path → Output
```

### Pipeline Stages

1. **Link Stage** (`link.py`): Merges Kineto (profiler) and Execution Trace (ET) data
2. **Prune Stage** (`link.py`): Removes spillover GPU kernels from previous iterations
3. **Build Dependencies** (`construct_dag.py`): Constructs dependency relationships between operations
4. **Create DAG** (`construct_dag.py`): Builds the full directed acyclic graph with all edge types
5. **Critical Path** (`dag.py`): Computes the longest path using topological sort and dynamic programming

---

## Input Files

### Kineto Trace (`kineto.json`)

Generated by PyTorch's profiler, contains:
- CPU operations (e.g., `aten::add`, `aten::batch_norm`)
- GPU kernels (e.g., `void at::native::vectorized_elementwise_kernel`)
- CUDA runtime calls (e.g., `cudaLaunchKernel`, `cudaDeviceSynchronize`)
- Timestamps (`ts`), durations (`dur`), process/thread IDs (`pid`/`tid`)
- Event categories (`cat`): `cpu_op`, `kernel`, `cuda_runtime`, etc.

### Execution Trace (`et.json`)

Generated by PyTorch's execution trace, contains:
- Node IDs and operation names
- Input/output tensor metadata (shapes, pointers, external IDs)
- Parent-child relationships between operations
- Record function IDs linking to Kineto events

### Linked Trace (`linked_trace.json`)

Generated by the linking stage, augments Kineto data with:
- `et_id`: Execution trace node ID
- `et_inputs`: List of input tensor pointers
- `et_outputs`: List of output tensor pointers
- `data_deps`: List of external IDs this operation depends on

### 2. `link.py` - Trace Linking and Preprocessing

#### Function: `process_files(kineto_file, et_file, linked_file)`

**Purpose**: Links Kineto profiler data with PyTorch execution trace data.

**Algorithm**:

1. **Load JSON Files**: Parse both Kineto and ET traces
2. **Build ET Lookup**: Map `attrs[0]['value']` → ET node ID
3. **Augment Kineto Nodes**: Add `et_id`, `et_inputs`, `et_outputs` to each Kineto event
4. **Build Tensor→Producer Map**: Track which operation produced each tensor
5. **Compute Data Dependencies**: For each operation, find producers of input tensors
6. **Write Linked Trace**: Save augmented Kineto data

**Data Dependency Calculation**:
```python
# Map tensor pointers to the operation that produced them
tensor_to_record_func_id = {}
for kin_node in kin_nodes:
    for tensor in kin_node.get("et_outputs", []):
        tensor_key = list_to_tuple(tensor)  # Convert to hashable
        tensor_to_record_func_id[tensor_key] = record_func_id

# For each operation, find dependencies
for kin_node in kin_nodes:
    data_deps = []
    for tensor in kin_node.get("et_inputs", []):
        tensor_key = list_to_tuple(tensor)
        producer_id = tensor_to_record_func_id.get(tensor_key)
        if producer_id and producer_id != current_record_func_id:
            data_deps.append(producer_id)
```

**Key Insight**: Data dependencies are established by matching input tensor pointers to the operations that produced them as outputs.

#### Function: `prune_spillover_kernels(events, debug=False)`

**Purpose**: Remove GPU kernels from previous training iterations that "spilled over" into the current trace window.

**Problem**: Traces often capture the tail end of GPU work from iteration N-1 before iteration N starts on the CPU.

**Algorithm**:

1. **Find Earliest Valid Kernel**: Identify the first GPU kernel that has a CPU parent in the call hierarchy
2. **Mark Spillover Kernels**: All GPU kernels with timestamps before the earliest valid kernel are spillovers
3. **Remove Spillover Events**: Delete spillover kernel events from the trace
4. **Clean References**: Remove spillover UIDs from `gpu_events` and `children` lists

**CPU Parent Detection**:
```python
# A GPU kernel has a CPU parent if its parent chain includes:
# cpu_op, cuda_runtime, or cuda_driver events
has_cpu_parent = False
parent_uid = event.get("parent")
while parent_uid and parent_uid != -1:
    parent_event = event_dict.get(parent_uid)
    if parent_event:
        parent_cat = parent_event.get("cat", "")
        if parent_cat in {"cpu_op", "cuda_runtime", "cuda_driver"}:
            has_cpu_parent = True
            break
```

**Why This Matters**: Spillover kernels create false dependencies and inflate the critical path by including work from a previous iteration.

### 3. `construct_dag.py` - Dependency Graph Construction

This module is the heart of the critical path analysis, implementing sophisticated dependency tracking.

#### Function: `build_dependency_map(tree1)`

**Purpose**: Build a temporal data dependency map that respects time ordering and prevents backward-in-time dependencies.

**The Tensor Reuse Problem**:
- PyTorch reuses tensor IDs across operations
- Naive dependency tracking creates spurious backward dependencies
- Example: If Op1 writes tensor T at t=100, and Op2 writes T at t=200, Op1 should NOT depend on Op2

**Three-Pass Algorithm**:

**Pass 1: Build Mappings**
```python
# Map external IDs to UIDs and collect timestamps
for event in tree1.events:
    external_id = event.get("args", {}).get("External id")
    uid = event.get("UID")
    ts = event.get("ts", 0)
    
    external_id_to_uid[external_id] = uid
    uid_to_external_id[uid] = external_id
    uid_to_timestamp[uid] = ts
```

**Pass 2: Track Last Writer**
```python
# For each tensor, track which operation last wrote to it
tensor_last_writer = {}  # external_id -> (uid, timestamp)

for event in tree1.events:
    external_id = event.get("args", {}).get("External id")
    ts = uid_to_timestamp.get(uid, 0)
    
    # Update if this is the first writer or a later writer
    if (external_id not in tensor_last_writer or 
        ts > tensor_last_writer[external_id][1]):
        tensor_last_writer[external_id] = (uid, ts)
```

**Pass 3: Build Dependencies with Temporal Ordering**
```python
for event in tree1.events:
    uid = event.get("UID")
    current_ts = uid_to_timestamp.get(uid, 0)
    data_deps = event.get("data_deps", [])
    
    for ext_id in data_deps:
        if ext_id == current_external_id:  # Skip self-reference
            continue
            
        if ext_id in tensor_last_writer:
            writer_uid, writer_ts = tensor_last_writer[ext_id]
            
            # Only create dependency if writer happened BEFORE
            if writer_ts < current_ts and writer_uid != uid:
                dependency_map[uid].append(writer_uid)
```

**Key Innovation**: By tracking the last writer by timestamp and only creating dependencies when `writer_ts < current_ts`, we eliminate all backward-in-time cycles.

#### Function: `create_dag(tree1, dependency_map, trace_start_uid=None, trace_end_uid=None, debug=False)`

**Purpose**: Construct the complete DAG with all dependency edge types.

**Edge Types** (defined in `EdgeType` enum):

1. **`DATA_DEPENDENCY`**: Producer-consumer relationships for tensors
   - Example: `matmul` depends on `transpose` that produced its input

2. **`CONTROL_DEPENDENCY`**: Parent-child relationships in the operation tree
   - Example: `aten::add` depends on its parent `aten::linear`

3. **`CPU_GPU_DEPENDENCY`**: CPU operations launching GPU kernels
   - Example: `cudaLaunchKernel` → GPU kernel execution
   - Only includes directly launched kernels, not entire subtree

4. **`GPU_CPU_DEPENDENCY`**: GPU completion before CPU synchronization
   - Example: Last GPU kernel → `hipDeviceSynchronize`
   - Adjusted latency accounts for wait time

5. **`INTRATHREAD_CPU_DEPENDENCY`**: Sequential CPU operations in same thread
   - Example: Op1 → Op2 where both are root operations in same thread
   - Maintains temporal ordering within threads

6. **`INTRATHREAD_GPU_DEPENDENCY`**: Sequential GPU kernels in same stream
   - Example: Kernel1 → Kernel2 in the same CUDA/HIP stream

7. **`TRACE_BOUNDARY`**: Artificial edges to source/sink nodes
   - Connects all root nodes to `trace_source`
   - Connects all leaf nodes to `trace_sink`

**DAG Construction Steps**:

1. **Initialize Nodes**: Create `DAGNode` for each event with basic attributes
2. **Add Data Dependencies**: Use `dependency_map` from temporal analysis
3. **Add Sequential CPU Dependencies**: Connect root operations within threads
4. **Add CPU→GPU Dependencies**: Link CPU ops to directly launched kernels
5. **Add Intrathread GPU Dependencies**: Connect kernels in same stream
6. **Add Device Sync Dependencies**: Handle blocking synchronization
7. **Add Source/Sink Nodes**: Create artificial boundary nodes
8. **Remove Duplicates**: Eliminate redundant edges and self-loops
9. **Remove Dangling Nodes**: Clean up disconnected subgraphs

#### Function: `_add_sequential_cpu_thread_dependencies(dag, tree1, debug=False)`

**Purpose**: Connect sequential root CPU operations within each thread.

**Root Operations**: Operations with `parent_dependencies == -1` (no parent in the tree).

**Algorithm**:
```python
# Group root operations by (pid, tid)
thread_root_ops = defaultdict(list)  # (pid, tid) -> [(ts, uid), ...]

for event in tree1.events:
    if dag_node.parent_dependencies == -1:
        pid = event.get("pid")
        tid = event.get("tid")
        ts = event.get("ts", 0)
        thread_root_ops[(pid, tid)].append((ts, uid))

# Sort by timestamp and connect consecutive operations
for thread_key, ops in thread_root_ops.items():
    ops.sort(key=lambda x: x[0])  # Sort by timestamp
    
    for i in range(len(ops) - 1):
        current_uid = ops[i][1]
        next_uid = ops[i + 1][1]
        
        # Add edge: current → next
        add_edge(current_uid, next_uid, EdgeType.INTRATHREAD_CPU_DEPENDENCY)
```

**Why This Matters**: Ensures thread-level temporal ordering is preserved, preventing false parallelism assumptions.

#### Function: `_add_cpu_gpu_dependencies(dag, tree1)`

**Purpose**: Link CPU operations to the GPU kernels they directly launch.

**The Subtree Problem**:
- Kineto stores all GPU kernels launched by a CPU subtree in `gpu_events`
- Naively linking all would create false dependencies
- Only the CPU op that directly calls `cudaLaunchKernel` should link to the kernel

**Solution: Call Hierarchy Tracing**:
```python
def _get_directly_launched_gpu_kernels(cpu_event, uid_to_event):
    """Get GPU kernels directly launched by this CPU op."""
    directly_launched = []
    
    # Find cuda_runtime children (cudaLaunchKernel, hipLaunchKernel, etc.)
    for child_uid in cpu_event.get("children", []):
        child = uid_to_event.get(child_uid)
        if child and child.get("cat") == "cuda_runtime":
            # Get GPU kernels launched by this cuda_runtime op
            for kernel_uid in child.get("gpu_events", []):
                directly_launched.append(kernel_uid)
    
    return directly_launched
```

**Call Hierarchy**:
```
CPU Op (e.g., aten::linear)
  └── cuda_runtime (cudaLaunchKernel)
       └── GPU Kernel (void at::native::vectorized_elementwise_kernel)
```

**Edge Creation**:
- CPU Op → GPU Kernel (only directly launched)
- Edge Type: `CPU_GPU_DEPENDENCY`

#### Function: `_add_device_sync_dependencies(dag, tree1, debug=False)`

**Purpose**: Model blocking CPU synchronization operations that wait for GPU work to complete.

**The Problem**:
- `hipDeviceSynchronize` / `cudaDeviceSynchronize` block until all GPU work finishes
- Measured duration includes:
  - Time GPU is still running (not CPU work)
  - Actual CPU overhead after GPU completes
- Naively using full duration inflates the critical path

**The Solution**:

1. **Find CPU Op Before Sync**: Look backward in time (same thread) for the last CPU op that launched a kernel
2. **Find Last Kernel**: Identify the latest-ending kernel launched by that CPU op
3. **Calculate Effective Sync Latency**: `sync_end - kernel_end` (only the wait time)
4. **Add GPU→CPU Dependency**: Create edge from kernel to sync with adjusted latency

**Algorithm**:
```python
for event in tree1.events:
    if "DeviceSynchronize" in event.get("name", ""):
        sync_uid = event.get("UID")
        sync_ts = event.get("ts", 0)
        sync_end = sync_ts + event.get("dur", 0)
        sync_tid = event.get("tid")
        
        # Find CPU ops before sync in same thread
        cpu_ops_before = [
            e for e in tree1.events
            if e.get("cat") == "cpu_op"
            and e.get("tid") == sync_tid
            and e.get("ts", 0) < sync_ts
        ]
        cpu_ops_before.sort(key=lambda x: x.get("ts", 0), reverse=True)
        
        # Find first CPU op with GPU children
        for cpu_op in cpu_ops_before:
            gpu_children = _get_directly_launched_gpu_kernels(cpu_op, uid_to_event)
            
            if gpu_children:
                # Find latest ending kernel
                latest_kernel_end = 0
                target_kernel = None
                for kernel_uid in gpu_children:
                    kernel_event = uid_to_event.get(kernel_uid)
                    kernel_end = kernel_event.get("ts") + kernel_event.get("dur")
                    if kernel_end > latest_kernel_end:
                        latest_kernel_end = kernel_end
                        target_kernel = kernel_event
                
                # Adjust sync latency
                effective_sync_latency = max(0, sync_end - latest_kernel_end)
                dag.update_node(sync_uid, measured_latency=effective_sync_latency)
                
                # Add GPU→CPU dependency
                add_edge(kernel_uid, sync_uid, EdgeType.GPU_CPU_DEPENDENCY)
                break
```

**Timeline Visualization**:
```
Time →
|---CPU Op---|
              |---cudaLaunchKernel---|
                                      |---GPU Kernel---|
                                                        |---hipDeviceSynchronize---|
                                                        ^                          ^
                                                        kernel_end              sync_end
                                                        |<-- effective_sync_latency -->|
```

**Why This Matters**:
- Prevents sync operations from dominating the critical path
- Correctly attributes time to the actual GPU work
- Models true blocking dependencies

#### Function: `_add_intrathread_gpu_dependencies(dag, tree1)`

**Purpose**: Connect GPU kernels executing sequentially in the same stream.

**GPU Stream Semantics**:
- Kernels in the same stream execute in submission order
- A kernel cannot start until the previous kernel in the stream completes

**Algorithm**:
```python
# Group GPU events by stream
stream_events = defaultdict(list)  # stream_id -> [(ts, uid), ...]

for event in tree1.events:
    if event.get("cat") in {"kernel", "gpu_memcpy", "gpu_memset"}:
        stream_id = event.get("args", {}).get("stream")
        ts = event.get("ts", 0)
        stream_events[stream_id].append((ts, uid))

# Sort by timestamp and connect consecutive kernels
for stream_id, events in stream_events.items():
    events.sort(key=lambda x: x[0])
    
    for i in range(len(events) - 1):
        current_uid = events[i][1]
        next_uid = events[i + 1][1]
        add_edge(current_uid, next_uid, EdgeType.INTRATHREAD_GPU_DEPENDENCY)
```

#### Function: `_add_source_and_sink_nodes(dag, tree1, start_uid=None, end_uid=None, debug=False)`

**Purpose**: Add artificial boundary nodes to ensure a single source and single sink.

**Why Needed**:
- DAG algorithms require single source/sink for critical path calculation
- Real traces have multiple entry points (different threads, operations)

**Source Node**:
```python
source_uid = "trace_source"
source_node = DAGNode(
    treenode_uid=source_uid,
    name="TRACE_SOURCE",
    category="trace_boundary",
    measured_latency=0,  # No latency
)
dag.add_node(source_node)

# Connect to all nodes with no input pointers
for node in dag.nodes:
    if not node.input_pointers:
        add_edge(source_uid, node.dag_uid, EdgeType.TRACE_BOUNDARY)
```

**Sink Node**:
```python
sink_uid = "trace_sink"
sink_node = DAGNode(
    treenode_uid=sink_uid,
    name="TRACE_SINK",
    category="trace_boundary",
    measured_latency=0,  # No latency
)
dag.add_node(sink_node)

# Connect from all leaf nodes (no output pointers)
for node in dag.nodes:
    if not node.output_pointers:
        add_edge(node.dag_uid, sink_uid, EdgeType.TRACE_BOUNDARY)
```

**Custom Start/End**:
- Can specify `start_uid` / `end_uid` to analyze partial traces
- Useful for zooming into specific operations

#### Function: `_remove_duplicate_dependencies(dag)`

**Purpose**: Clean up redundant edges and self-loops.

**Duplicates**: Same UID appearing multiple times in `input_pointers` / `output_pointers`

**Self-Loops**: Node pointing to itself (causes cycles)

**Algorithm**:
```python
for node in dag.nodes:
    # Remove duplicate inputs
    seen_inputs = set()
    new_input_pointers = []
    new_input_edge_types = []
    
    for uid, edge_type in zip(node.input_pointers, node.input_edge_types):
        if uid != node.dag_uid and uid not in seen_inputs:  # Not self-loop, not duplicate
            seen_inputs.add(uid)
            new_input_pointers.append(uid)
            new_input_edge_types.append(edge_type)
    
    node.input_pointers = new_input_pointers
    node.input_edge_types = new_input_edge_types
```

### 4. `dag.py` - Critical Path Computation

#### Class: `DAGNode`

**Attributes**:
- `dag_uid`: Unique identifier (same as `treenode_uid`)
- `treenode_uid`: UID from original trace event
- `name`: Operation name (e.g., `aten::linear`, `hipDeviceSynchronize`)
- `category`: Event category (`cpu_op`, `kernel`, `cuda_runtime`, etc.)
- `measured_latency`: Operation duration in microseconds
- `parent_dependencies`: Parent UID (control dependency)
- `target_latencies`: Target/expected latencies
- `input_pointers`: List of predecessor UIDs
- `output_pointers`: List of successor UIDs
- `input_edge_types`: List of `EdgeType` for each input edge
- `output_edge_types`: List of `EdgeType` for each output edge

**Methods**:
- `to_dict()`: Serialize to dictionary

#### Class: `DAG`

**Attributes**:
- `nodes`: List of all `DAGNode` objects
- `uid_to_dagnode`: Dictionary mapping UID → `DAGNode` for O(1) lookup

**Methods**:
- `add_node(node)`: Add node to DAG
- `get_node(uid)`: Retrieve node by UID
- `update_node(uid, **kwargs)`: Update node attributes
- `to_dict()`: Serialize entire DAG
- `find_critical_path()`: Compute longest path

#### Method: `find_critical_path()`

**Purpose**: Compute the critical path (longest path from source to sink) using topological sort and dynamic programming.

**Algorithm Overview**:

1. **Compute In/Out Degrees**: Count incoming and outgoing edges for each node
2. **Topological Sort (Kahn's Algorithm)**: Order nodes so all dependencies come before dependents
3. **Forward Pass (Longest Path)**: Compute longest path to each node
4. **Backtrack**: Reconstruct path by following longest predecessors

**Detailed Algorithm**:

**Step 1: Compute Degrees**
```python
in_degree = defaultdict(int)
out_degree = defaultdict(int)

for node in self.nodes:
    out_degree[node.dag_uid] = len(node.output_pointers)
    for neighbor in node.output_pointers:
        in_degree[neighbor] += 1
```

**Step 2: Topological Sort (Kahn's Algorithm)**
```python
queue = deque()
topological_order = []

# Initialize with source nodes (in-degree 0)
for node in self.nodes:
    if in_degree[node.dag_uid] == 0:
        queue.append(node.dag_uid)

while queue:
    current = queue.popleft()
    topological_order.append(current)
    
    current_node = self.get_node(current)
    for neighbor in current_node.output_pointers:
        in_degree[neighbor] -= 1
        if in_degree[neighbor] == 0:
            queue.append(neighbor)
```

**Cycle Detection**:
```python
if len(topological_order) != len(self.nodes):
    # Cycle detected!
    nodes_in_cycles = set(node.dag_uid for node in self.nodes) - set(topological_order)
    # Log nodes involved in cycles
```

**Step 3: Forward Pass - Longest Path Calculation**

**CRITICAL INITIALIZATION**:
```python
# Initialize ALL nodes to 0 (not their own latency!)
longest_path = {node.dag_uid: 0 for node in self.nodes}
predecessor = {}
```

**Why Start at 0?**
- Common mistake: Initialize `longest_path[uid] = measured_latency`
- This prevents updates when predecessor paths are longer
- Example: If node has latency 10, initialized to 10, but predecessor has path 100, update fails (100 + 10 > 10)
- Correct: Start at 0, add latency as edge weight

**Forward Pass Logic**:
```python
for current_uid in topological_order:
    current_node = self.get_node(current_uid)
    current_path = longest_path[current_uid]
    
    # Update all successors
    for i, neighbor_uid in enumerate(current_node.output_pointers):
        edge_type = current_node.output_edge_types[i]
        
        # Determine edge weight based on edge type
        if edge_type == EdgeType.TRACE_BOUNDARY:
            weight = 0  # No latency for artificial edges
        else:
            weight = current_node.measured_latency  # Use node's latency
        
        new_path = current_path + weight
        
        # Update if this path is longer
        if new_path > longest_path[neighbor_uid]:
            longest_path[neighbor_uid] = new_path
            predecessor[neighbor_uid] = current_uid
```

**Edge Weight Rules**:
- `TRACE_BOUNDARY`: 0 (artificial edges to source/sink)
- All other edges: Use `measured_latency` of the source node
- For `GPU_CPU_DEPENDENCY`: Uses adjusted sync latency (calculated in `_add_device_sync_dependencies`)

**Step 4: Backtrack to Reconstruct Path**
```python
# Find sink node (should be "trace_sink")
sink_uid = "trace_sink"
critical_path = []

current = sink_uid
while current in predecessor:
    critical_path.append(current)
    current = predecessor[current]

critical_path.reverse()

# Remove artificial boundary nodes
critical_path = [uid for uid in critical_path 
                 if uid not in {"trace_source", "trace_sink"}]
```

**Logging**:
- Writes detailed debug log to `critical_path_debug.log`
- Includes topological order, longest paths, predecessor choices
- Useful for debugging dependency issues

**Top 5 Longest Paths**:
```python
# Find top 5 longest paths (excluding source/sink)
path_lengths = [(uid, length) for uid, length in longest_path.items()
                if uid not in {"trace_source", "trace_sink"}]
path_lengths.sort(key=lambda x: x[1], reverse=True)

for uid, length in path_lengths[:5]:
    node = self.get_node(uid)
    print(f"  {node.name} (UID:{uid}): {length:.3f} µs")
```

## Algorithm Details

### Critical Path Computation Complexity

- **Time Complexity**: O(V + E)
  - V = number of nodes
  - E = number of edges
  - Topological sort: O(V + E)
  - Longest path: O(V + E)
  - Backtracking: O(V)

- **Space Complexity**: O(V + E)
  - Store all nodes and edges
  - Auxiliary structures for in-degrees, longest paths

### Handling Cycles

**Cycle Prevention Strategy**:

1. **Temporal Ordering**: Data dependencies only created when `writer_ts < reader_ts`
2. **No Self-Loops**: Nodes cannot depend on themselves
3. **Duplicate Removal**: Eliminate redundant edges

**If Cycles Are Detected**:
- Logged to `critical_path_debug.log`
- Lists all nodes involved in cycles
- Critical path computation stops
- Must fix dependency construction logic

### Edge Weight Semantics

Different edge types contribute differently to the critical path:

| Edge Type | Weight | Rationale |
|-----------|--------|-----------|
| `DATA_DEPENDENCY` | `measured_latency` | Producer must finish before consumer |
| `CONTROL_DEPENDENCY` | `measured_latency` | Parent must finish before child |
| `CPU_GPU_DEPENDENCY` | `measured_latency` | CPU op duration before kernel launch |
| `GPU_CPU_DEPENDENCY` | `adjusted_latency` | Only wait time after GPU finishes |
| `INTRATHREAD_CPU_DEPENDENCY` | `measured_latency` | CPU ops execute sequentially |
| `INTRATHREAD_GPU_DEPENDENCY` | `measured_latency` | GPU kernels in stream execute sequentially |
| `TRACE_BOUNDARY` | `0` | Artificial edges, no real work |

### Spillover Kernel Detection

**Why Spillovers Happen**:
- Profiler captures a time window
- Training iteration N-1 may still have GPU work running when iteration N starts
- These "spillover" kernels are not part of the current iteration

**Detection Method**:
1. Find earliest GPU kernel that has a CPU parent in the call hierarchy
2. All GPU kernels before this timestamp are spillovers
3. Remove spillover kernels and references

**Impact**:
- Typical traces have 2,000-3,000 spillover kernels
- Removing them prevents false critical path inflation

## Troubleshooting

### Issue: No Critical Path Found

**Symptoms**: Critical path is empty or only contains source/sink.

**Possible Causes**:
1. Cycles in the DAG (check `critical_path_debug.log`)
2. All nodes pruned as spillovers
3. Incorrect trace format

**Solutions**:
1. Enable `--debug` to see pruning details
2. Check `nodes_in_cycles` in debug log
3. Verify trace files are valid JSON

### Issue: Critical Path Too Short

**Symptoms**: Critical path has only a few nodes.

**Possible Causes**:
1. Longest path initialization bug (should be 0, not `measured_latency`)
2. Missing dependency types
3. Incorrect edge weights

**Solutions**:
1. Check `longest_path` initialization in `find_critical_path()`
2. Verify all `_add_*_dependencies()` functions are called
3. Review edge weight logic for each `EdgeType`

### Issue: Device Sync Dominates Path

**Symptoms**: `hipDeviceSynchronize` / `cudaDeviceSynchronize` has very long latency on critical path.

**Solution**: Ensure `_add_device_sync_dependencies()` is called and adjusts sync latency to only include wait time after GPU work completes.

### Issue: Spillover Kernels Not Removed

**Symptoms**: Many GPU kernels with no CPU parent on critical path.

**Solution**: Verify `prune_spillover_kernels()` is called before DAG construction and correctly identifies CPU parents in the call hierarchy.

## Performance Considerations

### Trace Size

- **Small Traces** (< 10,000 nodes): Process in seconds
- **Medium Traces** (10,000 - 100,000 nodes): Process in 10-30 seconds
- **Large Traces** (> 100,000 nodes): May take minutes, consider filtering

### Memory Usage

- **Rule of Thumb**: ~1 KB per node
- 100,000 nodes ≈ 100 MB memory
- Linked traces are larger due to ET metadata

### Optimization Tips

1. **Use Pre-Linked Traces**: Skip linking step if analyzing same trace multiple times
2. **Prune Spillovers**: Significantly reduces graph size
3. **Filter Categories**: Only analyze relevant operation types
