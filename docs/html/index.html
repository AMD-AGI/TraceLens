TraceLens

An AMD open-source library for automating GPU performance analysis from profiler trace files — turning raw traces into structured roofline analysis, kernel summaries, and communication reports.

Python ≥ 3.6 PyTorch · JAX · rocprofv3 · Perfetto AMD ROCm Copyright 2024–2025 AMD

01Introduction

What TraceLens is, why it exists, and what problems it solves.

TraceLens is a Python library and CLI toolkit built by AMD for automated GPU performance analysis. It reads profiler trace files — output from PyTorch's built-in profiler, JAX's XPlane protobuf format, ROCm's rocprofv3, or Google's Perfetto — and produces structured Excel/CSV reports with roofline analysis, per-op performance metrics, NCCL collective analysis, and trace comparison.

The core insight is that raw trace files are unwieldy: they contain millions of events from CPU threads, GPU kernels, HIP runtime calls, and NCCL communication — all interleaved. TraceLens's job is to reconstruct the logical computation tree, attach GPU kernels to the CPU operations that launched them, apply analytical performance models (FLOPs, memory bytes, roofline bounds), and produce human-readable reports.

What It Solves

🌲
Tree reconstruction
Converts flat timestamp-ordered events into a nested CPU call-stack tree with GPU kernels attached via correlation IDs.
📐
Roofline analysis
62+ performance model classes compute theoretical FLOPs and memory bytes for GEMM, SDPA, Conv, RMSNorm, MoE, and more. Compares measured vs. peak hardware throughput.
📡
Collective analysis
Multi-rank NCCL/RCCL collective analysis with bus bandwidth, skew metrics, and synchronization detection across all GPU ranks.
🔀
Trace comparison
Morphological diff of two traces via tree edit distance (Wagner-Fischer), plus column-level Excel comparison of two perf reports.
🧩
Extensible system
A three-part plugin system (tree_postprocess, perf_model, dict_cat2names) lets users inject custom ops, perf models, and categories without forking the library.
🔄
Operation replay
EventReplayer re-executes operations from trace metadata via PyTorch JIT, enabling isolated benchmarking of individual ops.

02Quick Start

Install and generate your first report in under 5 minutes.

Installation

# Clone and install
git clone https://github.com/amd/tracelens
cd tracelens
pip install -e .

# Optional: comparative analysis using slodels with LLM backends
pip install -e ".[comparative]"

Generate a PyTorch Performance Report (CLI)

TraceLens_generate_perf_report_pytorch \
    --profile_json_path  path/to/trace.json.gz \
    --output_xlsx_path   report.xlsx \
    --output_csvs_dir    ./report_csvs/ \
    --arch               mi300x

Generate a PyTorch Performance Report (Python API)

from TraceLens import TreePerfAnalyzer

# Load trace, build tree, attach performance models
analyzer = TreePerfAnalyzer.from_file(
    profile_filepath="trace.json.gz",
    arch="mi300x",
)

# GPU timeline breakdown (compute / comm / memcpy / idle)
df_timeline = analyzer.get_df_gpu_timeline()

# Full unified performance table
df_perf = analyzer.build_df_unified_perf_table()

# Summary by op category
df_summary = analyzer.summarize_df_unified_perf_table()
print(df_summary)

Compare Two Traces

TraceLens_compare_perf_reports_pytorch \
    --baseline_xlsx  baseline.xlsx \
    --variant_xlsx   variant.xlsx \
    --output_xlsx    comparison.xlsx

Multi-rank NCCL Analysis

TraceLens_generate_multi_rank_collective_report_pytorch \
    --trace_dir  ./rank_traces/ \
    --output_xlsx nccl_report.xlsx
Tip — Architecture JSON

The --arch argument selects a hardware config. TraceLens ships with presets (e.g. mi300x). You can also pass a path to a custom JSON file with fields: name, freq_mhz, num_cus, mem_bw_gbps, l1_bw_gbps, peak TFLOPS per dtype.

Step-by-step Breakdown

  1. Load — DataLoader.load_data(path) reads the compressed JSON using orjson for performance. The result is a list of raw trace event dicts.
  2. Categorize events — TraceEventUtils separates events by ph (phase) field: X (complete events), B/E (begin/end pairs), M (metadata), s/f (flow events for forward/backward linking).
  3. Build tree — TraceToTree.build_host_call_stack_tree() sorts CPU events by timestamp and constructs a parent-child nesting based on temporal containment. Four layers: Python frontend → PyTorch op / cpu_op → HIP/CUDA runtime → GPU kernel.
  4. Link GPU kernels — add_gpu_ops_to_tree() correlates each GPU kernel to its CPU launcher via the correlation or External id field in args.
  5. Link fwd/bwd — link_all_fwd_bwd_events() uses autograd flow events (ph: 's'/'f') to associate forward ops with their backward counterparts.
  6. Inject pseudo-ops — apply_pseudo_op_extensions(tree) scans for known fused kernel patterns (MoE, MLA decode, etc.) and inserts synthetic CPU ops so individual kernels get their own roofline entries.
  7. Attach perf models — TreePerfAnalyzer.compute_perf_metrics(event, arch) looks up the op name in op_to_perf_model_class_map, instantiates the right perf model class, and calls .flops() / .bytes().
  8. Compute timeline — GPUEventAnalyser merges overlapping GPU kernel intervals using interval arithmetic to compute: computation, exposed communication, exposed memcpy, busy, and idle time.
  9. Build DataFrames — build_df_unified_perf_table() collects all annotated tree nodes into a flat DataFrame with 50+ columns covering timing, shapes, FLOPs, bytes, kernel details, NCCL options.
  10. Write report — generate_perf_report_pytorch() writes multiple sheets to an .xlsx file and optionally per-sheet CSVs.

06Trace2Tree

TraceLens/Trace2Tree/trace_to_tree.py · 1218 lines · Converts flat trace events into a hierarchical tree.

The tree structure is the foundation of TraceLens's analysis. Instead of working with millions of flat events, TraceLens reconstructs the logical call stack that was executing when each GPU kernel ran — so performance data can be attributed to the right PyTorch op, nn.Module, or user-annotated region.

Tree Layer Model

LayerCategoryExample Events
Layer 0Python frontend / user codemodel.forward(), custom annotations
Layer 1PyTorch op / cpu_opaten::linear, aten::scaled_dot_product_attention
Layer 2HIP/CUDA runtimehipLaunchKernel, cudaLaunchKernel
Layer 3 (leaves)GPU kernelsvoid gemm_kernel<...>, NCCL ops

BaseTraceToTree (Abstract)

ABSTRACT CLASSBaseTraceToTree
  • build_host_call_stack_tree() — Sorts CPU events by ts, then uses a stack to detect parent-child containment. A child event's [ts, ts+dur] interval falls within its parent's interval.
  • label_non_gpu_paths() — Post-order traversal marking nodes that have no GPU kernel descendants (used to skip pure-CPU paths in reporting).

TraceToTree (PyTorch)

CLASSTraceToTree(BaseTraceToTree)
  • add_gpu_ops_to_tree() — Builds a map from correlation / External id → GPU kernel. Walks the CPU tree; when a runtime call has a matching correlation ID, attaches the GPU kernel as a gpu_children entry on the nearest CPU op ancestor.
  • link_all_fwd_bwd_events() — Reads autograd flow events (ph: 's' = fwd start, ph: 'f' = bwd finish) to link forward PyTorch ops to their corresponding backward ops. Enables attribution of backward GPU time to the originating forward operation.
  • apply_annotation(annotation_fn) — Traverses the tree and calls a user function on each node to set the annotation field. Used by the inference analysis pipeline.
  • traverse_subtree_and_print(node, depth) — Debug utility that pretty-prints the tree.

JaxTraceToTree

CLASSJaxTraceToTree(BaseTraceToTree)
JAX variant that uses correlation_id linking from the XPlane protobuf format. Integrates with JaxProfileProcessor for HLO-level metadata.

Graph Capture / Replay Merging

trace_capture_merge_experimental.py (573 lines) handles the vLLM/SGLang graph capture pattern where CUDA/HIP graph capture and replay produce different event layouts. It detects dummy-run patterns and merges graph capture events with their replay counterparts so the analysis is coherent.

How Pseudo-Ops Work

inject_pseudo_op(parent, kernel, name, extra_args) inserts a new synthetic node between a parent CPU op and a GPU kernel child. The synthetic node carries metadata (MoE type, topk, gated flag, etc.) that the perf model uses to compute FLOPs and bytes. apply_pseudo_op_extensions(tree) auto-detects all four patterns and applies the appropriate injection.

GPUEventAnalyser

TraceLens/TreePerf/gpu_event_analyser.py · 508 lines · Computes GPU utilization via interval arithmetic.

CLASSGPUEventAnalyser
  • merge_intervals(intervals) → list — Merges overlapping [start, end] intervals into a minimal set of non-overlapping intervals. O(n log n).
  • subtract_intervalsA_from_B(A, B) → list — Set-difference: returns the parts of B not covered by any interval in A.
  • get_gpu_event_lists() — Separates all GPU events into three lists: computation, communication (NCCL/RCCL), memcpy.
  • compute_metrics_dict() → dict — Applies merge + subtract to produce: computation, exposed_comm, exposed_memcpy, busy, idle, total in microseconds.
Interval Arithmetic

Exposed communication = comm intervals that don't overlap compute. Exposed memcpy = memcpy intervals not overlapped by compute or exposed comm. This gives the true serialized overhead, not the raw collective time.

08PerfModel — Analytical Performance Models

TraceLens/PerfModel/perf_model.py · 5355 lines · 62 classes providing FLOPs, memory bytes, and roofline predictions for GPU operations.

Each perf model class corresponds to one or more PyTorch ops (e.g., aten::mm, aten::scaled_dot_product_attention). Given the op's tensor shapes (from the trace's Concrete Inputs field), it computes:

  • FLOPs — Floating-point operations (arithmetic count)
  • Bytes — HBM bytes read and written
  • Roofline bound — Whether the op is compute-bound or memory-bandwidth-bound given the architecture
  • Simulation time (optional) — For GEMM/SDPA: subprocess-based or Origami-based hardware simulation

Model Families

1. GEMM Family

Base class for all matrix multiply operations. flops_func(M, N, K, bias) = 2·M·N·K + (M·N if bias). Bytes = read A + read B + optional bias read + write output.

2. CONV Family

Supports Conv1d/2d/3d, transposed convolutions, and fused bias/ReLU variants. Output shape computed analytically from kernel size, stride, padding, dilation. Forward parameters cached for use by backward pass via fwd_pass_cache.

3. Softmax

flops_func(M, N, B) = B·M·3·N (exp + sum + div for each of N elements across M rows). Uses roofline to distinguish L1-resident vs HBM-bound cases.

4. SDPA Family — Scaled Dot Product Attention

The most complex model family, covering all flash attention variants.

PV_flops = 2 * B * H_Q * N_Q * N_KV * d_h_v    # P·V
total_fwd = QK_flops + PV_flops
# For causal: halved (upper triangle masked)
total_fwd_causal = total_fwd / 2

Implicit Synchronization Detection

TraceLens detects implicit synchronization — when one rank starts a collective late, forcing other ranks to wait even though no explicit barrier exists. This manifests as high arrival-time skew across ranks for the same collective instance. The build_df_nccl_implicit_sync_cat() method quantifies this.

15Origami Integration

TraceLens/PerfModel/origami_helper.py · 408 lines · ROCm Origami hardware simulator integration.

By default, TraceLens uses analytical formulas for FLOPs and bytes. With --enable_origami, it instead calls ROCm Origami — a hardware simulation tool — for GEMM and SDPA predictions.

CLASSOrigamiHelper
  • get_hardware(arch) → OrigamiHardware — Creates an Origami hardware config from the architecture JSON (reads freq_mhz, num_cus, mem_bw_gbps, etc.)
  • get_simulation_time(op_type, params) → float — Returns predicted kernel execution time in microseconds. Requires rocm-origami package and a running ROCm environment.
Requirements

Origami requires pip install rocm-origami (not in the default dependencies) and an active ROCm runtime. Without it, TraceLens falls back to analytical models silently.

17Extension System

A three-part plugin mechanism for customizing TraceLens without forking.

Pass --extension_file path/to/my_extension.py to any report generator. The file may define any or all of three hooks:

1. tree_postprocess_extension

def tree_postprocess_extension(trace_to_tree):
    # Receives the fully-built TraceToTree object
    # Inject pseudo-ops, rename nodes, or add metadata
    from TraceLens.Trace2Tree.extensions.pseudo_ops_utils import inject_pseudo_op
    for node in find_nodes(trace_to_tree.root, "my_fused_op"):
        inject_pseudo_op(node, node.gpu_children[0], "my_pseudo_up", {})
        inject_pseudo_op(node, node.gpu_children[1], "my_pseudo_down", {})

2. perf_model_extension

# Dict mapping op name strings to perf model classes
perf_model_extension = {
    "my_pseudo_up": MyUpProjPerfModel,
    "my_pseudo_down": MyDownProjPerfModel,
    "te_layer_norm_fwd": LayerNorm,
}

3. dict_cat2names_extension

# Dict mapping category names to lists of base class names
dict_cat2names_extension = {
    "MyCustomOp": ["MyUpProjPerfModel", "MyDownProjPerfModel"],
}
Example: Megatron-LM Extension

examples/example_megatron_extension.py ships a complete working extension that adds TransformerEngine layer norm (te_layer_norm_fwd/bwd) perf models and links gradient checkpoint forward/backward passes.
